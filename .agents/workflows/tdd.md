# Test-Driven Development (TDD) Prompt & Rules

이 프로젝트에서 TDD(테스트 주도 개발) 방식으로 기능을 구현하거나 AI(Antigravity 등)에게 작업을 지시할 때 베이스가 되는 프롬프트 및 컨벤션입니다.

## 🎯 AI에게 지시할 TDD 시스템 프롬프트

AI 어시스턴트에게 새로운 기능을 개발해달라고 요청할 때, 아래의 프롬프트를 함께 전달하거나 이 파일의 룰을 따르라고 지시하세요:

> "앞으로 요구하는 모든 기능은 반드시 **TDD(Test-Driven Development)** 방법론을 엄격하게 준수하여 구현해 주세요. 아래의 5가지 핵심 규칙을 완벽하게 지켜야 합니다.
> 
> 1. **Red, Green, Refactor 사이클 준수**
>    - **Red**: 실제 로직을 구현하기 전에, 요구사항을 검증하는 실패하는 테스트 코드를 먼저 작성하세요.
>    - **Green**: 오직 해당 테스트를 통과하기 위한 목적만으로, 가장 간결하고 단순한 프로덕션 코드를 작성하세요.
>    - **Refactor**: 테스트가 통과한 상태를 유지하면서, 코드의 가독성을 높이고 중복을 제거하며 클린 아키텍처 구조에 맞게 리팩토링하세요.
> 
> 2. **3가지 수준의 다층적 테스트 작성**
>    - 도메인 로직 및 개별 함수/컴포넌트를 검증하는 **Unit Test (단위 테스트)**
>    - 계층 간 상호작용(예: 유스케이스와 리포지토리, UI와 API)을 검증하는 **Integration Test (통합 테스트)**
>    - 실제 사용자 관점에서 브라우저 플로우를 검증하는 **E2E Test (End-to-End 테스트)**
>    - 위 3가지 테스트를 요구사항에 맞게 모두 작성해야 합니다.
> 
> 3. **Coverage 80% 이상 유지**
>    - 새로 작성되거나 수정된 파일의 테스트 커버리지(Statement, Branch, Function, Line)는 항상 **80% 이상**을 유지하도록 테스트 시나리오를 구성하세요.
> 
> 4. **전체 테스트 재실행 및 최종 확인**
>    - 모든 구현 및 리팩토링 작업이 끝난 후에는 반드시 전체 테스트 스위트를 다시 실행하여, 기존 기능에 사이드 이펙트가 발생하지 않았는지 모든 테스트 통과 여부를 확인해 주세요.
> 
> 5. **실패 원인 분석 및 자가 수정**
>    - 테스트 재실행 시 통과하지 못하는 테스트가 단 하나라도 존재한다면, **왜 실패했는지 근본적인 원인을 상세히 분석**하여 리포트하고, 즉시 코드를 수정하여 초록색(Pass) 상태로 복구해 주세요."

## 🛠 Next.js & Supabase 환경에서의 적용 팁
* **Unit / Integration**: 로직이 분리된 도메인/애플리케이션 계층은 의존성 없이 순수 TypeScript 환경(Jest 또는 Vitest)에서 빠르게 테스트합니다. React 컴포넌트는 `React Testing Library`를 활용합니다.
* **E2E**: `Playwright` 또는 `Cypress`를 활용하여 회원가입, 대시보드 지표 확인 등의 주요 렌더링 및 클릭 흐름을 테스트합니다.
* **의존성 모킹(Mocking)**: 테스트 속도를 위해 DB(Supabase)와 통신하는 Infrastructure 계층의 Repository는 테스트 시 Mock 객체로 대체하여 Integration Test를 수행합니다.
